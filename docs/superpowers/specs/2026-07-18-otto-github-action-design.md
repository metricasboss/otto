# OTTO Phase 2a — Engine Hardening + GitHub Action (Design)

**Date:** 2026-07-18
**Status:** Approved for planning
**Depends on:** Phase 1 engine (merged at `b334fee`), spec `2026-07-17-otto-privacy-review-engine-design.md`

## Goal

Ship the visible half of the Rams thesis: a GitHub Action that reviews every PR's changed files with the Phase 1 engine, posts a sticky comment with the deterministic 0-100 privacy score and per-finding details (LGPD/GDPR article, fine exposure, suggested fix), uploads SARIF for inline annotations, and fails the check below a configurable gate. Before that, close the four hardening gaps the Phase 1 final review identified.

**Out of scope (Phase 2b+):** LLM contextual pass, MCP server, hosted SaaS, marketplace listing polish.

## Part 1 — Engine hardening (pre-work)

1. **MultiEdit hook bypass.** `otto/hook.py` scans only `new_string`/`content`; a MultiEdit-shaped payload (`tool_input.edits[].new_string`) passes unscanned. Fix: concatenate all `edits[].new_string` values (joined with `"\n"`) into the scanned content alongside the existing fields. Test: MultiEdit payload with a check-digit-valid CPF → deny.
2. **SARIF path relativization.** `otto/engine/reporters/sarif.py` emits the path as given; absolute paths don't map in GitHub Code Scanning. Fix: if the finding's path is inside the current working directory, emit it relative to CWD (POSIX separators); otherwise leave it unchanged. Test: scan via absolute path under CWD → relative `uri`.
3. **Per-rule case sensitivity.** New optional schema field `"case_sensitive": true` (default false). Scanner omits `re.IGNORECASE` for such rules (keeps `re.MULTILINE`). Apply to `national_id_eu`, whose `[A-Z]{2}\d{6,9}[A-Z]?` currently matches lowercase git SHAs like `'ab1234567'`. Fixture pair updated accordingly (clean fixture may now include a lowercase near-miss). Harness covers it automatically.
4. **Installer cleanup.** `install.sh`: remove legacy `$skills_dir/scripts/` on upgrade; run the legacy PostToolUse cleanup even when `--no-hooks` is passed (cleanup is not hook installation). Bump `actions/checkout` and `actions/setup-python` in `.github/workflows/test.yml` to the current majors (Node 20 deprecation warning).

## Part 2 — GitHub Action

### Packaging

Composite action, `action.yml` at the root of `metricasboss/otto`. Consumers use `uses: metricasboss/otto@v2` (or `@main`). No separate repo, no JS build; the engine ships inside the action checkout.

### Inputs

| input | default | meaning |
|---|---|---|
| `fail-under` | `60` | job fails if PR score < N; `0` disables the gate |
| `regulation` | `both` | `lgpd` \| `gdpr` \| `both` |
| `comment` | `true` | post/update the sticky PR comment |
| `sarif` | `true` | upload SARIF to Code Scanning |

### Required consumer workflow

```yaml
on: pull_request
permissions:
  contents: read
  pull-requests: write
  security-events: write
jobs:
  privacy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
        with: { fetch-depth: 0 }
      - uses: metricasboss/otto@v2
        with:
          fail-under: 60
```

### Step flow (inside the composite)

1. **Changed files:** `git diff --name-only --diff-filter=d "origin/$GITHUB_BASE_REF...HEAD"` in the consumer's checkout (requires `fetch-depth: 0`; the action fetches the base ref itself if missing). Deleted files are filtered (the CLI exits 2 on missing paths — correct behavior, so they must not reach it). File list written to a temp file and passed via `xargs`-safe invocation to avoid arg-length limits. Zero files after filtering → score 100 comment ("nothing to analyze"), job passes, no SARIF.
2. **Scan:** `python3 -m otto scan <files> --regulation <input> --format json > otto-report.json` and a second run with `--format sarif > otto.sarif`, both with `--fail-under 0` (the gate is a separate, final step so reporting always happens first). Python set up via `actions/setup-python` (3.12); no pip installs (stdlib-only engine).
3. **Sticky comment:** `actions/github-script` step. Finds an existing PR comment containing the marker `<!-- otto-privacy-report -->` and updates it, else creates it. Body is generated by a small Python module (see Components) from `otto-report.json`: headline score `N/100` (with the critical-cap note when applicable), findings table (`file:line`, severity, rule, article, fine, suggested fix), and a footer stating the score is deterministic and linking the repo. If `comment: false`, skip. If the API call fails (fork PR read-only token), emit a workflow warning and continue — never fail the job because commenting failed.
4. **SARIF upload:** `github/codeql-action/upload-sarif` with `continue-on-error: true` (private repos without GHAS, fork PRs). Skipped when `sarif: false` or when there were zero scanned files.
5. **Gate:** final step re-runs the CLI with `--fail-under <input>` (cheap, deterministic) — its exit code is the job result. Runs after reporting so a failing PR still shows why.

### Components

- `action.yml` — composite definition (inputs, steps above).
- `otto/action/report.py` — pure function `build_comment(report: dict) -> str` (JSON report in, markdown comment out, marker included) + `python3 -m otto.action.report otto-report.json` CLI shim used by the comment step. All comment-formatting logic lives here so it is pytest-testable; the YAML stays thin.
- `.github/workflows/otto.yml` — dogfooding: the OTTO repo runs its own action (`uses: ./`) on every PR.

### Edge cases

- **Fork PRs:** comment and SARIF upload may be denied (read-only token) → warnings, not failures; the gate still enforces. Documented limitation in README.
- **Base ref not fetched:** action runs `git fetch origin $GITHUB_BASE_REF` before diffing.
- **Non-PR trigger (push):** action fails fast with a clear error — v1 supports `pull_request` events only.
- **Files outside supported types:** engine glob rules already exclude markdown/tests per rule; no extra filtering in the action.

### Documentation

README gains a "GitHub Action" section: the consumer workflow snippet, inputs table, permissions explanation, fork-PR limitation, and a screenshot-placeholder-free description of the comment format.

## Success criteria

- Hardening: MultiEdit payload with valid CPF denied by hook; SARIF `uri`s relative for in-CWD scans; `'ab1234567'` no longer flagged by `national_id_eu` while `'AB1234567'` still is; reinstall leaves no `$skills_dir/scripts/`; full suite green.
- A PR against the OTTO repo (dogfooding workflow) that adds a file with a valid CPF gets: a sticky comment with score ≤59 citing LGPD Art. 46, and a failed check; pushing a fix updates the same comment (no duplicates) and the check passes.
- A clean PR gets a 100/100 comment and a passing check.
- `build_comment` covered by pytest (score headline, table rows, marker, empty-findings variant).
- No new runtime dependencies; comment logic is stdlib-only Python.
